## Deep Dive: Roles & ARIA Attributes â€” Relationship Explained

### Short Answer
Yes. **ARIA roles and ARIA attributes are part of the same system.** Both belong to **WAI-ARIA**.

But:
- **Role = what the element *is***
- **ARIA attributes = what the element *is doing* or its *state* or *relationships***

They work *together*, but they are NOT the same thing.


### Analogy: â€œRole = noun, ARIA attributes = adjectivesâ€

Example:
- ğŸ”˜ Button (role)
- ğŸ”˜ Expanded = true (aria-expanded)
- ğŸ”˜ Has popup (aria-haspopup)

Here: role tells *what you are*; aria-* tells *what state you are in*.


### Examples

#### Example 1: Dropdown Menu

Semantic HTML:

```html
<button aria-expanded="false">Menu</button>
```

Here: implicit `role="button"`. `aria-expanded` tells state.

Custom component:

```html
<div role="button" aria-expanded="false" tabindex="0">
  Menu
</div>
```

`role="button"` tells assistive tech: â€œThis is a button.â€
`aria-expanded="false"` tells: â€œDropdown is closed.â€


#### Example 2: Modal/Dialog

Custom modal:

```html
<div role="dialog" aria-modal="true" aria-labelledby="title">
  <h2 id="title">Confirmation</h2>
</div>
```

- `role="dialog"` identifies the component
- `aria-modal="true"` tells screen reader to trap focus
- `aria-labelledby="title"` tells the name of dialog


### What Are ARIA Roles Exactly?

ARIA roles are labels that give meaning to elements, especially non-semantic ones.

Types:

1. **Widget roles** (interactive): `role="button"`, `role="checkbox"`, `role="switch"`, `role="dialog"`, `role="combobox"`, `role="menu"`, `role="tab"`
2. **Structure roles**: `role="banner"`, `role="navigation"`, `role="main"`, `role="contentinfo"`
3. **Document roles**: `role="article"`, `role="paragraph"`

Many roles match semantic HTML tags (implicit roles). This is why you donâ€™t need ARIA on semantic HTML.


### What Are ARIA Attributes?

ARIA attributes describe:

- **State** â€” `aria-checked`, `aria-expanded`, `aria-selected`, `aria-disabled`
- **Property** â€” `aria-haspopup`, `aria-controls`, `aria-labelledby`, `aria-describedby`
- **Hidden or live regions** â€” `aria-hidden`, `aria-live`
- **Names** â€” `aria-label`


### How Roles and ARIA Attributes Work Together

Different roles expect different ARIA attributes. Roles define what attributes are allowed/valid.

- Buttons can use: `aria-pressed`, `aria-expanded`, `aria-haspopup`
- Checkbox expects: `role="checkbox"` + `aria-checked="true/false"`
- Tabs expect: `role="tab"`, `aria-selected`, `aria-controls`
- Dialog expects: `role="dialog"`, `aria-modal`


### Analogy
Think of ARIA like a LinkedIn profile.
- **Role = your job title** ("Software Engineer")
- **ARIA attributes = your job details** ("Remote", "Available")

The role tells the main identity. ARIA adds extra information.


### Final Summary of Roles & Attributes

- âœ”ï¸ Are ARIA roles and ARIA attributes related? **Yes.**
- âœ”ï¸ Roles define: **What is this element?**
- âœ”ï¸ ARIA attributes define: **What state is it in?**
- âœ”ï¸ They work together, especially in custom components.

## Accessibility Patterns & Code Examples

Below are a few common widget patterns and complete accessible examples.

### Accessible Button (native)

```html
<button id="save">Save</button>
```

- No ARIA needed.
- Screen readers announce: "Save, button".


### Icon-only Button

```html
<button aria-label="Close dialog">
  <svg ...>...</svg>
</button>
```

- Use `aria-label` because visible text is not present.


### Accessible Custom Button (div)

```html
<div role="button" tabindex="0" aria-pressed="false" id="like">
  <span class="icon-like" aria-hidden="true"></span>
  <span class="visually-hidden">Like</span>
</div>
```

- `role` gives identity
- `tabindex` makes it focusable
- Use `aria-pressed` to reflect state
- Provide an accessible name via visible or hidden text


### Accessible Modal (outline)

```html
<!-- Trigger -->
<button id="open">Open modal</button>

<!-- Modal structure -->
<div id="modal" role="dialog" aria-modal="true" aria-labelledby="modal-title" hidden>
  <h2 id="modal-title">Modal Title</h2>
  <p>Modal content...</p>
  <button id="close">Close</button>
</div>

<!-- Focus management notes (JS):
1. Save element that opened the modal.
2. Move focus to first focusable element in modal on open.
3. Trap focus inside modal.
4. On close, restore focus to opener.
-->
```


### Accessible Tabs (outline)

```html
<div role="tablist" aria-label="Sample Tabs">
  <button role="tab" aria-selected="true" aria-controls="panel-1" id="tab-1">Tab 1</button>
  <button role="tab" aria-selected="false" aria-controls="panel-2" id="tab-2">Tab 2</button>
</div>

<div id="panel-1" role="tabpanel" aria-labelledby="tab-1">Content 1</div>
<div id="panel-2" role="tabpanel" aria-labelledby="tab-2" hidden>Content 2</div>
```


### Live Region Example

```html
<div aria-live="polite" id="cart-announce" aria-atomic="true"></div>
<script>
  function announce(message) {
    const node = document.getElementById('cart-announce');
    node.textContent = message; // screen readers will announce
  }
</script>
```


## Testing Checklist (Practical)

1. Tab through your whole app (keyboard only)
2. Check focus outlines are visible
3. Try activating controls with Enter/Space
4. Test screen reader (VoiceOver / NVDA)
5. Run Lighthouse / axe DevTools
6. Check color contrast (4.5:1 / 3:1 large text)
7. Zoom to 200% and confirm layout works
8. Test live updates using `aria-live`
9. Test that modals trap focus and restore it
10. Ensure there are no keyboard traps


## Common Mistakes to Avoid

- Using `div` or `span` for interactive elements without role/tabindex
- Overusing ARIA attributes when semantic HTML exists
- Wrong role+attribute combinations (e.g., `button` with `role="link"`)
- Not managing focus on dialogs/modals
- Hiding content with CSS but not `aria-hidden` when required
- Removing focus outline without adding a visible replacement
